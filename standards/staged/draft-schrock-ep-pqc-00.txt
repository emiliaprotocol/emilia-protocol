        EMILIA Protocol: Crypto-Agility and Post-Quantum Profile
                        draft-schrock-ep-pqc-00

Abstract

   This document defines a crypto-agility framework and post-quantum
   (PQC) signature profile for EMILIA Protocol (EP) authorization
   receipts ([I-D.schrock-ep-authorization-receipts]).  EP receipts are
   long-lived (multi-year regulatory retention), so signature and digest
   algorithms MUST be explicitly identified and migratable.  This
   profile specifies an algorithm registry, a hybrid (classical plus
   PQC) signature mode for a no-flag-day migration, and a fail-closed
   verifier accepted-algorithm policy.  The hybrid signing path is a
   forward roadmap; EP today signs Ed25519 over RFC 8785 (JCS) canonical
   bytes, and EP Evidence-Record already renews digests across algorithm
   aging.

1.  Algorithm Registry

   Receipts and signoffs carry an explicit algorithm identifier.
   Defined classes: Ed25519 (EdDSA) and ECDSA P-256 (ES256) for
   classical/device signatures; ML-DSA (FIPS 204) and SLH-DSA (FIPS 205)
   for post-quantum signatures; and SHA-256 to SHA-384/512 (FIPS 180-4)
   for digest agility.  ML-KEM (FIPS 203) is key encapsulation, not
   signatures, and is out of scope for the authorization signature.

2.  Hybrid Mode (the migration path)

   A hybrid signature carries both a classical (Ed25519) and a PQC (ML-
   DSA) signature over the same JCS-canonical payload ([RFC8785]); under
   hybrid policy a receipt verifies only if BOTH verify.  This gives
   backward verifiability (classical-only verifiers still check),
   forward security (the PQC part carries weight once "harvest-now,
   decrypt-later" matters), and no flag day (issuers add the PQC
   signature; verifiers upgrade independently).  The signature object
   names the algorithm, e.g.  hybrid-ed25519+ml-dsa-65, with separate
   classical and pqc values.

3.  Verifier Policy

   A verifier declares an accepted-algorithm policy: classical-only
   (today), hybrid-required (transition), or pqc-required (post-
   migration).  Fail-closed: an algorithm not in policy is a rejection,
   never a silent downgrade.

4.  Relationship to Other Work

   This profile composes with EP Evidence-Record (long-lived receipts
   re-anchored under stronger algorithms as they age).  National-
   algorithm jurisdictions (for example SM2/SM3 and their PQC
   adaptations) register in the same algorithm registry -- same agility
   mechanism, different identifiers.  It aligns in direction with the
   NIST PQC standards and the NSA CNSA 2.0 migration timeline.

5.  Security Considerations

   Algorithm downgrade is the dominant risk; the verifier accepted-
   algorithm policy is fail-closed by design, and hybrid mode ensures a
   single broken family does not invalidate a receipt.  PQC signing MUST
   use a vetted implementation; this profile specifies no novel scheme.

6.  IANA Considerations

   This document has no IANA actions; algorithm identifiers register in
   the EP profile registry.
