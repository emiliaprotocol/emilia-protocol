EMILIA Protocol

The human-authorization receipt layer for agentic AI  ·  proposed for AAIF TC review  ·  non-binding feedback sought

MCP governs how an agent connects. goose governs how it executes. AGENTS.md governs how it's guided. EMILIA answers the question those leave open: when an agent's action is irreversible — a payment, a vendor-record change, a deletion — what artifact proves a named human authorized that exact action before it ran?

EMILIAproof a human authorized the irreversible action

Decision logs are testimony. Receipts are evidence.

What it is

An offline-verifiable authorization receipt: a device-bound human signoff (Ed25519 over RFC 8785 canonical JSON) over the exact action — verifiable by anyone with open-source code, with no backend and no vendor trust. The MCP middleware intercepts an irreversible tool call and returns 428 — no receipt, no execution until a valid, exact-action receipt is presented. Change one field of the action and the signature fails; replay it and it's refused.

It's real, and it's small

  • IETF Internet-Draft — draft-schrock-ep-authorization-receipts (posted on the IETF datatracker).
  • Reference verifiers in JavaScript, Python, and Go — byte-identical agreement across a shared conformance vector suite.
  • Formally-verified core — 26 TLA+ safety properties, 0 errors; Alloy models for the quorum path.
  • MCP middleware that enforces the loop, plus SDKs (@emilia-protocol/require-receipt, /gate, /verify).
  • 60-second check, no account or backend:  npx @emilia-protocol/issue demo
