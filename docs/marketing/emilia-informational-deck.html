The shift

Software stopped asking permission.
It just acts now.

AI agents already release payments, push deploys, change permissions, curtail datacenters, and command robots — at machine speed, with no human in the loop. The actions are consequential and often irreversible. The question every regulator, insurer, and board now asks is the one no one can answer: who authorized that, and can you prove it?
01 · Why this mattersEMILIA Protocol
The gap

"A human approved this"
is a claim no one can check.

Today that proof lives in the operator's own editable logs — forgeable, backfillable, and un-auditable by a regulator, court, or insurer without trusting the very party under review. Logs tell you what happened. They cannot prove who authorized it. That evidence gap is a category — and it is unfilled.
02 · The problemoversight is mandated everywhere, verifiable nowhere
What EMILIA is

Automated control for
every high-risk agent action.

Think passport control for AI agents. Before an agent may take a consequential action, EMILIA checks it: is there a valid authorization tracing to a named human who approved this exact action? If yes — AUTHORIZED, proceed. If not — DENIED. EMILIA is not the agent's identity and not the tool it calls. It is the checkpoint and the stamp.
Authorized
named human authorizer present · signed this exact action → proceed
Denied
authorizer unavailable / out of scope → action does not execute

04 · What it producesEd25519 · RFC 8785 (JCS) · offline · one-time
How it works

Five steps. Deny-by-default until proven.

1
Agent requests a consequential action. EMILIA intercepts at the actuator boundary and returns 428 — authorization required.
2
A named human signs the exact action on their own device (Face ID / passkey). The signature is bound to a hash of the specific action, not a blanket approval.
3
EMILIA verifies the receipt: valid signature, in-scope, sufficiently assured, not replayed.
4
The action executes — only now.
5
An execution receipt is emitted — portable proof it ran, verifiable by anyone, offline.
05 · The flowrequest → 428 → human signs → verify → execute → receipt
The product

EMILIA Gate —
the Consequence Firewall.

Deny-by-default at the actuator boundary. A consequential action runs only with a valid, in-scope, sufficiently-assured, non-replayed receipt — then emits proof it ran. A new security category: not antivirus (is this file bad?), not a firewall (is this packet allowed?) — is this machine allowed to take this consequence, and can you prove who authorized it?
Software — MCP tool calls, agent runtimes, CI/CD, systems of record.
Cloud — API-level enforcement in front of any consequential endpoint.
Physical — robots and actuators: no release without a verified receipt.
