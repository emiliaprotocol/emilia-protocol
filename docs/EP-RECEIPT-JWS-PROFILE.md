# EP-RECEIPT-JWS-PROFILE-v1 — a JWS serialization of EP authorization receipts

**Status:** Profile (spec-level). Defines an OPTIONAL JSON Web Signature
[RFC7515] serialization so an EMILIA authorization receipt can be consumed by
any standard JOSE verifier with no EP-specific code. Not a wire-format mandate:
the canonical EP artifact remains the JSON `EP-RECEIPT-v1` document with its
detached Ed25519 signature in [draft-schrock-ep-authorization-receipts].

## Why this exists

EP's native receipt is JSON canonicalized with JCS [RFC8785] and signed with a
**detached** Ed25519 [RFC8032] signature carried in `doc.signature.value`. That
is the right shape for the agent-authorization receipt cluster, but it is *not*
a shape an off-the-shelf JOSE library can verify directly. A huge installed base
of verifiers, gateways, and policy engines already speak JWS. The highest-
leverage interop bet is therefore to let any standard JOSE verifier (e.g. the
`jose` npm library) consume an EP receipt. This profile re-signs the **same
canonical receipt material** as a compact JWS, with zero change to EP semantics.

## The envelope

An `EP-RECEIPT-v1` document is serialized as a **JWS Compact Serialization**
[RFC7515 §3.1, §7.1]:

```
BASE64URL(UTF8(protected header)) '.' BASE64URL(JWS payload) '.' BASE64URL(signature)
```

### Protected header

| Member | Value | Reference |
|---|---|---|
| `alg`  | `"EdDSA"` (Ed25519) | [RFC8037 §3.1] |
| `typ`  | `"application/ep-receipt+jws"` | [RFC7515 §4.1.9] |
| `kid`  | OPTIONAL key id (see below) | [RFC7515 §4.1.4] |

`alg` MUST be `EdDSA` over an Ed25519 key. EP keys are the same base64url
SPKI-DER Ed25519 public keys used everywhere else in EP. No other `alg` is
permitted in this profile (in particular `none` and any symmetric MAC are
forbidden — consistent with EP's no-symmetric-signing invariant).

`kid`, when present, is the first 16 bytes of `SHA-256(base64url SPKI public
key)`, hex-encoded — a deterministic, key-derived fingerprint. It is purely
advisory: verification trusts the key the relying party supplies / pins, never
the `kid`. EP receipts have no native `kid`, so this is generated by the
profile when an issuer public key is known.

### JWS payload

The JWS payload is the **JCS [RFC8785] canonical bytes of the receipt
`payload`** — byte-for-byte the same material the native EP-RECEIPT-v1 signature
covers. (EP's canonicalization is recursive depth-first key-sorted JSON over an
I-JSON value subset: strings, booleans, null, arrays, objects, and safe
integers; non-integer reals MUST be string-encoded. See
[`packages/verify`](../packages/verify/index.js).)

### Signature

`EdDSA` (Ed25519, [RFC8032]) over `ASCII(BASE64URL(header) '.'
BASE64URL(payload))` per [RFC7515 §5.1].

> **This is a PARALLEL envelope, not the native signature.** The JWS signature
> value is NOT equal to `doc.signature.value`. The native EP signature is over
> the canonical payload bytes directly; the JWS signature is over
> `b64u(header).b64u(payload)`. They are two independent signatures over the
> same canonical receipt material; both authenticate the exact same receipt. A
> consumer MAY verify either or both.

## Verification steps

A verifier (EP-native or any JOSE library) MUST:

1. Split the compact serialization into exactly three non-empty segments and
   decode the protected header from segment 1.
2. Enforce the profile header: `alg === "EdDSA"` and
   `typ === "application/ep-receipt+jws"`. Reject otherwise.
3. Verify the Ed25519 signature over `ASCII(b64u(header) '.' b64u(payload))`
   against the issuer's (pinned/trusted) Ed25519 public key.
4. Parse the payload bytes back to the receipt `payload` object.
5. **Round-trip check:** `JCS(parsedPayload)` MUST equal the verified payload
   bytes byte-for-byte. This rejects any non-canonical re-encoding and
   guarantees the JWS authenticated the EP canonical form. Reject on mismatch.

Verification fails closed: any structural, header, signature, or round-trip
mismatch is a rejection, and the verifier never throws on attacker-controlled
input.

A standard JOSE verifier naturally performs steps 1–4; step 5 is the EP-specific
binding that ties the JWS back to the canonical EP receipt. EP's reference
implementation (`verifyReceiptJws`) performs all five.

## Reference implementation

- `serializeReceiptJws(doc, privateKey, opts)` and
  `verifyReceiptJws(jws, publicKey)` —
  [`packages/require-receipt/jws.js`](../packages/require-receipt/jws.js),
  re-exported from `@emilia-protocol/require-receipt`. Runtime path uses only
  Node's built-in `crypto`; `jose` is a dev-only cross-verification dependency.
- Conformance vectors:
  [`conformance/vectors/jws.json`](../conformance/vectors/jws.json)
  (regenerate byte-identically with `node conformance/vectors/generate-jws.mjs`).
- Cross-verification with the standard `jose` library is proven in
  [`packages/require-receipt/jws.test.js`](../packages/require-receipt/jws.test.js):
  `jose.compactVerify` accepts the produced JWS and rejects a tampered one.

## What it does not change

Semantics are identical: same JCS canonical payload, same Ed25519 keys, same
fail-closed verification. The JSON `EP-RECEIPT-v1` form remains canonical and
normative; this is an alternate envelope for JOSE interop, selected by media
type `application/ep-receipt+jws`. The native detached signature is unaffected.
