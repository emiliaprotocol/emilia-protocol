# EP-ENVELOPE-v1 — the statement envelope + profile registry

*Experimental / additive over the FROZEN EP-RECEIPT-v1. Governed by PIP-012. Not a
production or customer claim; reports no metrics. Implementation:
`lib/envelope/` · registry: `public/.well-known/ep-profiles.json`.*

## Why this exists
EMILIA shipped a frozen receipt core and a family of additive profiles (revocation,
WYSIWYS, execution-integrity, eye-set continuous-eval, provenance chain), but each
arrived as its own wire shape and its own `verifyX()` with a different signature.
That makes the system read as a pile of tools: adding capability N+1 requires being
EMILIA. EP-ENVELOPE-v1 is the **narrow waist** that turns the profiles — and the
consequential actions they cover — into **data**: one wire object, one verifier
that dispatches by a registered profile URN, and one composition rule. A third
party registers a profile in a reserved namespace **without a core change**.

## The envelope
```
{
  "ep":      "EP-ENVELOPE-v1",
  "profile": "urn:ep:profile:<name>:v<n>",          // resolved against the registry
  "typ":     "<optional media-type hint>",
  "payload": { ... },                                // the profile-specific body
  "binding": { "action_hash": "...", "policy_hash": "...", "prev_hash": "..." },  // optional, uniform
  "proofs":  [ { "algorithm": "Ed25519", "kid": "...", ... } ],                   // optional, envelope-level
  "anchor":  { ... },                                // optional transparency anchor
  "meta":    { ... }
}
```

## Profile URNs
- **Core:** `urn:ep:profile:<name>:v<n>` (registered by EMILIA / the standard).
- **Reserved private-use:** `urn:ep:profile:x-<vendor>:<name>` — **anyone may ship one today, without coordination.** The core never collides with it.

## Verification (`verifyEnvelope(env, opts)`)
A shared, profile-agnostic pipeline runs first and **no profile can skip it**:
1. `envelope_version` — `ep` MUST be `EP-ENVELOPE-v1`.
2. `profile_known` — `profile` MUST be a well-formed URN **and registered**. An unknown profile fails closed.
3. `payload_present` — `payload` MUST be a non-null object.
4. `proof_alg_allowed` — any envelope-level proof MUST use an allowed algorithm; `none` and unlisted algorithms are rejected.

Then the registered profile's `validateBody(env, opts)` runs and returns
`{ valid, checks, errors }`. The verdict is:

> **`valid = sharedOk && body.valid` — the PluginCannotWeaken invariant.**

A profile plugin can therefore only **add** rejections; it can NEVER turn a shared
`valid:false` into `true`, and a plugin that throws is treated as a rejection (the
verifier never crashes on adversarial input). This is the security guarantee that
makes third-party profiles safe to register: an untrusted plugin cannot make a
structurally-invalid envelope verify, and cannot affect any other profile.

## Wrapped vs native profiles (honest scope)
The five built-in profiles are **wrapped**: their `validateBody` adapts the uniform
envelope to the profile's existing verifier (which already fails closed, e.g.
rejecting unpinned signer keys). The envelope adds the shared pipeline + the
composition guarantee on top; the inner verifier still performs the profile-specific
cryptography. A **native** profile expresses its checks directly against the shared
pipeline. Both are first-class; wrapping is the migration path, not a permanent
seam.

## `migrate()` — lossless, no re-signing
`migrate(obj, urn)` wraps a previously-issued profile object as
`{ ep, profile, payload: obj }` such that `canonicalize(payload) === canonicalize(obj)`
byte-for-byte. Existing receipts and the posted IETF I-D stay valid; the envelope is
purely additive over them.

## The registry (`public/.well-known/ep-profiles.json`)
A content-addressed manifest: each profile row carries a `content_hash`
(sha256 over its canonical descriptor) and the manifest a `registry_hash`, so a
consumer verifies the registry **offline** and detects tampering. EMILIA hosts a
**mirror, not the root of trust** — profiles are self-described and (for vendor
profiles) self-published. `public/.well-known/ep-actions.json` is the offline
vocabulary of consequential action types a profile can cover; new action types are
new rows, not core changes. Both are regenerated by
`node scripts/build-ep-registry.mjs` from the single descriptor source
(`lib/envelope/descriptors.js`); `tests/ep-registry.test.js` proves they never
drift.

## Adding a profile
`node scripts/create-ep-profile.mjs --vendor <slug> --name <slug>` scaffolds a
plugin (<50 lines) + a conformance-vectors stub. Implement `validateBody` (fail
closed; verify signatures only under a verifier-pinned key), register it, regenerate
the registry. Cross-profile conformance comes for free from the shared verifier +
the manifest-checked vectors.

## Residual / out of scope
The envelope unifies the **outer** shape, dispatch, and composition guarantee; it
does not re-implement each profile's cryptography (wrapped profiles delegate inward).
It does not by itself create demand — a profile is only *required* where a relying
party (or a standard like an MCP SEP that references a profile URN) demands a valid
envelope. Self-publishing a vendor profile asserts authorship via the content hash;
binding that hash to a real-world identity is a transparency/PKI concern layered
above this spec.
