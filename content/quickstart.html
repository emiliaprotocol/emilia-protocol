Quick Start Guide

Protect Your First Workflow in 30 Minutes

The core EP trust loop: register a policy, initiate a handshake, present credentials, verify, and gate an action.

By the end of this guide, you will have a working trust-protected workflow. Five steps. All curl commands. No SDK required.

Prerequisites: An EP API key (starts with ep_live_ or ep_test_), two registered entities (your agent and a counterparty), and curl or any HTTP client.
Shell — Setup
# Set your API key once
$ export EP_KEY="ep_live_your_key_here"
$ export EP_BASE="https://emiliaprotocol.ai"
Local development: Replace with http://localhost:3000 as your base URL.

Step 1 — Browse Available Policies

1
Understand what trust policies are available
~2 minutes

Before initiating a handshake, understand what trust policies are available. Policies define the thresholds for trust decisions.

Shell
$ curl "$EP_BASE/api/policies"

The response includes available policies with their requirements:

standard
Normal transactions. Balanced requirements for confidence and evidence. min_score: 40, min_confidence: provisional
strict
High-value transactions. Requires established confidence, low dispute rate. min_score: 70, min_confidence: confident
Pick a policy that matches your risk level. For this guide, we use "standard". Policy families include commerce, software, marketplace, and custom.

Step 2 — Initiate a Handshake

2
Structured identity exchange before a trust decision
~5 minutes

A handshake is a structured identity exchange between two parties before a trust decision is made. You declare who is involved, which policy governs, and what mode of exchange to use.

Shell — Initiate handshake
$ curl -X POST "$EP_BASE/api/handshake" \
  -H "Authorization: Bearer $EP_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "mode": "mutual",
    "policy_id": "standard",
    "parties": [
      { "entity_ref": "your-agent-entity-id", "role": "initiator" },
      { "entity_ref": "counterparty-entity-id", "role": "responder" }
    ]
  }'

Save the handshake_id from the 201 Created response. You will use it in every subsequent step.

Shell — Save the handshake ID
$ export HS_ID="hs_abc123def456"
Error cases: 400 -- missing or invalid fields. 401 -- missing or invalid API key. 403 -- initiator entity_ref does not match the authenticated entity.

Step 3 — Present Credentials

3
Each party presents identity claims for evaluation
~5 minutes

Each party presents identity claims for evaluation. The initiator presents first.

Shell — Present credentials (initiator)
$ curl -X POST "$EP_BASE/api/handshake/$HS_ID/present" \
  -H "Authorization: Bearer $EP_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "party_role": "initiator",
    "presentation_type": "ep_trust_profile",
    "claims": {
      "entity_id": "your-agent-entity-id",
      "entity_type": "agent",
      "display_name": "My Agent"
    },
    "disclosure_mode": "full"
  }'

The responder does the same with "party_role": "responder".

Presentation types: ep_trust_profile (EP trust profile), verifiable_credential (W3C VC), attestation (third-party attestation). Disclosure modes: full, selective, zk (zero-knowledge proof).

Step 4 — Verify the Handshake

4
Evaluate the handshake against the governing policy
~2 minutes

Once all parties have presented, evaluate the handshake against the governing policy. This is the trust decision.

Shell — Verify handshake
$ curl -X POST "$EP_BASE/api/handshake/$HS_ID/verify" \
  -H "Authorization: Bearer $EP_KEY" \
  -H "Content-Type: application/json" \
  -d '{}'

# Response — Accepted
{
  "handshake_id": "hs_abc123def456",
  "result": "accepted",
  "reason_codes": [],
  "evaluated_at": "2026-03-20T10:02:00Z"
}
Possible results: accepted -- both parties meet policy thresholds. rejected -- with specific reason_codes like insufficient_confidence or dispute_rate_exceeds_policy. partial -- still awaiting presentations from one or more parties.

Step 5 — Gate the Action

5
Authorize a specific action with a trust-backed commit
~5 minutes

After a successful handshake, use the trust gate to authorize a specific action. The gate evaluates the entity, checks the handshake binding, and issues a commit -- a signed pre-authorization token.

Shell — Gate the action
$ curl -X POST "$EP_BASE/api/trust/gate" \
  -H "Authorization: Bearer $EP_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "entity_id": "counterparty-entity-id",
    "action": "transact",
    "policy": "standard",
    "handshake_id": "hs_abc123def456",
    "value_usd": 500
  }'

# Response — Allowed
{
  "decision": "allow",
  "entity_id": "counterparty-entity-id",
  "policy_used": "standard",
  "confidence": "emerging",
  "reasons": [],
  "appeal_path": "https://emiliaprotocol.ai/appeal",
  "extensions": {
    "action": "transact",
    "handshake_verified": true,
    "commit_ref": "cmt_abc123"
  }
}

The commit_ref is your proof that EP authorized this action. Store it. Every denial includes an appeal_path -- this is a constitutional guarantee.

Constitutional principle: Trust must never be more powerful than appeal. Every adverse decision is explainable, challengeable, and reversible when wrong.

What Comes Next

After completing the trust loop, extend your integration with these capabilities:

  • Submit a receipt after the action completes: POST /api/receipts/submit
  • Revoke a handshake if something goes wrong: POST /api/handshake/{id}/revoke
  • File a dispute if the counterparty misbehaves: POST /api/disputes/file
  • Check trust profile of any entity: GET /api/trust/profile/{entityId}

SDKs

Reference SDKs for TypeScript and Python. Both wrap the REST API with typed interfaces.

Shell
# TypeScript / Node.js
$ npm install @emilia-protocol/sdk

# Python
$ pip install emilia-protocol

SDK source is in sdks/ in the main repo. Both SDKs are Apache 2.0.
