// Why It Matters
Trust broke. Machines inherited the wreckage.
Stars, reviews, and download counts don't scale into a world where AI agents make decisions at machine speed. The trust systems we have were built for humans — and they were already failing.

No Portable Trust

A merchant's reputation on Amazon means nothing on Shopify. A plugin rated safe on npm isn't recognized on the MCP registry. Every ecosystem reinvents the wheel and every user starts from zero.

Real cost: Sellers lose years of earned reputation on migration. Agents accept dangerous defaults because no portable signal exists.

No Context

A generic 4.2-star score doesn't tell you if a seller is safe for a $5,000 electronics purchase. A plugin safe for read-only access is dangerous with code execution. Generic trust is dangerous trust.

Real cost: Agents over-trust in high-risk contexts. Platforms set thresholds that make no sense for their specific use case.

No Due Process

Platform trust scores change silently. Appeals go unanswered. Merchants lose income without explanation. Developers watch packages flagged with no recourse. If trust can hurt you, it must be challengeable.

Real cost: Trust becomes a weapon. Competitors file false reports. Platforms abuse the power of silence.

// How It Works
Receipts, Not Reviews
Every legacy trust system failed because the platform hosting the data profits from the businesses being evaluated. EMILIA is different. The buying agent submits a receipt about the seller — and then uses that same trust profile to decide where to transact next. You rate what you eat. Lie, and you eat bad food again. The submitter and the consumer of trust data are the same entity. This creates natural incentive alignment that no legacy trust system has achieved.
1
Entity Registers
Agents, merchants, and service providers register on the protocol. Each gets a permanent entity ID and API key. Registration takes one API call.
2
Transactions Happen
Commerce flows through existing protocols — UCP, ACP, A2A, MCP. EP doesn't mediate the transaction. It evaluates whether the counterparty should be trusted next time.
3
Receipts Are Submitted
The buying agent submits a receipt: transaction_ref, behavioral outcome (completed/retried/abandoned/disputed), optional context keys, and structured claims. Four-factor weighting: submitter credibility × time decay × graph health × provenance tier.
4
Trust Recomputes
The trust profile recomputes from rolling receipts. Behavioral-first weights with effective-evidence dampening. Fully auditable. No one can buy their trust — unestablished submitters carry only 0.1× weight.
// The Three Phases of Truth
From Self-Reported to Protocol-Observed
PHASE 1 LIVE

Behavioral Truth

Agents submit receipts with behavioral outcomes as the primary signal (40% weight). Did they come back? Switch? Abandon? Four-factor receipt weighting resists Sybil attacks from day one.

PHASE 2 IN DEV

Evidence-Backed Truth

Agents submit structured claims with evidence. "Delivered 3 days late" + tracking ID. "Price overcharged 5%" + payment ref. Conflicting claims enter disputed state — dispute rate becomes a signal.

PHASE 3 ROADMAP

Protocol-Observed Truth

EP listens to MCP/A2A/UCP protocol events and auto-generates receipts. Decentralized verification oracles (with their own EP trust profiles) confirm claims against carrier APIs. Zero-knowledge proofs for privacy.

// Questions
Frequently Asked
How is EP different from Trustpilot, Yelp, or Amazon ratings?
EP is structurally different in three ways: the rater has skin in the game (the buying agent uses the same trust profiles it submits), the algorithm is fully published (any skeptic can verify the weights), and every decision is appealable (trust can never be more powerful than appeal). Trustpilot and Yelp are reviewer-controlled, platform-profiting, and opaque. EP is behavioral, auditable, and constitutionally bounded.
Is EP a blockchain project? Do I need tokens or a wallet?
No tokens, no wallet, no DeFi. EP uses blockchain anchoring (Base L2 Merkle roots) purely for auditability — so anyone can verify that a trust record hasn't been tampered with. The protocol itself is a REST API and MCP server. No crypto knowledge required.
Can someone game the trust system by submitting fake receipts?
EP has four layers of Sybil resistance: (1) Submitter credibility — unestablished entities carry only 0.1× weight regardless of what they claim. (2) Graph health — closed-loop submitter networks are detected and penalized (0.4×). (3) Cluster collusion — coordinated artificial inflation reduces weight to 0.1× and triggers block. (4) Effective-evidence dampening — volume of receipts doesn't matter, only the credibility-weighted evidence does. The system is designed so that faking trust at scale is more expensive than earning it honestly.
What happens when a trust profile is wrong about me?
You appeal. The appeal page requires no account and no login. You can file a report about inaccurate profiles, dispute specific receipts with evidence, and appeal dispute resolutions to a second reviewer. The constitutional principle is: trust must never be more powerful than appeal. EP is designed so that every negative trust effect is explainable, challengeable, and correctable without erasing history.
Is EP free? What's the business model?
The protocol is Apache 2.0 open source — free to use, fork, and implement. The hosted API at emiliaprotocol.ai is free during public beta. Future commercial plans will cover enterprise SLAs, dedicated operator support, and custom policy management. Self-hosting is always free. EP will never charge for the core trust evaluation primitives.
What's "install preflight" and why do I need it?
Install preflight (ep_install_preflight or POST /api/trust/install-preflight) answers: should I install this plugin/app/package? It evaluates a software entity against a software-specific trust policy covering: publisher identity verification, permission class analysis (read-only vs. code execution vs. network access), behavioral trust history, and provenance. Returns allow, review, or deny with specific reasons. Use it before any AI agent installs an MCP server, GitHub App, npm package, or browser extension.
What are trust policies? How are they different from scores?
A trust score is a number (4.2 stars). A trust policy is a decision framework: "pass if confidence is established AND completion rate > 85% AND dispute rate < 5% in the electronics category". EP ships four built-in policies (strict, standard, permissive, discovery) plus four software policies. You can also pass custom policy logic as JSONB. Policies are context-aware — electronics and apparel can have different requirements. They return pass/fail with specific failure reasons, not just a number.
How does EP handle identity continuity? What if a merchant re-registers?
EP-IX (Identity Continuity Extension) prevents trust whitewashing. Every entity is backed by a principal — the enduring actor (person, org, agent system) behind the entity IDs. When an entity re-registers, continuity must be declared and challenged. Predecessors, successors, and continuity decisions are publicly tracked. A bad actor can't cheaply escape their history by creating a new account — EP-IX makes trust portable for good actors while making history persistent for bad ones.
Can I run my own EP instance? How do I become an operator?
Yes — EP is fully self-hostable under Apache 2.0. Run npm install && npm run dev to start locally. For production, deploy to Vercel or any Node.js host with a Supabase/PostgreSQL database. To become a dispute resolution operator — the humans who review and resolve appeals — see the Operators page. Operators are accountable entities with their own EP trust profiles. Their performance (response time, appeal outcomes, error rate) is publicly tracked.
How does EP relate to NIST, AAIF, and AI standards?
EP has submitted engagement materials to NIST ITL (National Institute of Standards and Technology Information Technology Laboratory) and is proposing an AAIF (Agent-to-Agent Interoperability Forum) working group for a formal agent trust standard. EP is designed to be the reference implementation for what a W3C/IETF "Agent Trust Standard" could look like. The spec is fully published and has a conformance suite with 147 automated tests — anyone can implement EP and verify conformance.

// Adversarial Resilience
What happens when a bad actor tries to game the system?
EP was designed assuming adversarial conditions from day one. Here is how the protocol responds to the most common manipulation attempts.
Attack
Flood receipts with fake positive reviews
A Sybil network submits hundreds of receipts attesting high quality for a malicious entity.
// Protocol response
Submitter credibility weight applied (0.1× for new submitters). Graph health detects closed-loop patterns. Effective evidence dampening ensures fake volume does not create real confidence. Cluster collusion detection flags coordinated networks.
Attack
File false reports to damage a competitor
A bad actor floods the report queue with malicious dispute filings against a legitimate entity.
// Protocol response
Rate limiting (3/hr per submitter fingerprint). Report credibility weighted by submitter history. Operator review required before any trust change. Dismissed reports lower the submitter's own credibility score.
Attack
Whitewash identity after reputation damage
An entity with a damaged trust profile re-registers under a new name or entity ID.
// Protocol response
EP-IX identity lineage tracking links successor entities to predecessors. Host verifier attestations bind canonical identities. Prior receipt graph follows the principal, not just the name.
Attack
Bribe or coerce an operator decision
A well-resourced attacker attempts to influence an operator to reverse a legitimate uphold.
// Protocol response
All operator reasoning is public and permanent. Baseline comparison flags divergence. Governance removal process is peer-triggered, not platform-controlled. No single operator can be the only reviewer.
Attack
Submit receipts for transactions that never happened
An entity submits self-generated receipts to bootstrap a fake trust profile with no real interactions.
// Protocol response
Bilateral receipts require both parties to submit. Provenance tiers weight host-verified receipts (1.0×) over self-asserted (0.4×). Unilateral volume never reaches PROVISIONAL confidence without corroboration.
Attack
Silently alter trust records after anchor
An attacker gains database access and modifies historical receipts to erase negative history.
// Protocol response
SHA-256 chain links every record to its predecessor. Merkle roots anchored to Base L2 are immutable. Any modification is detectable by anyone with the anchor hash — no trust required.
// Evidence Requirements
From zero to confident
Trust isn't binary. EP tracks five confidence levels based on effective evidence — not raw receipt counts. Fake volume cannot advance an entity past INSUFFICIENT.
Level
Requirements & Meaning
Agent Behavior
PENDING
Entity registered but no receipts submitted. Effective evidence = 0.
Reject or require manual override
INSUFFICIENT
Receipts exist but effective evidence < 3.0. Volume may be high but submitter credibility and graph health are low.
High-risk: apply conservative policy
PROVISIONAL
Effective evidence ≥ 3.0. Some credible receipts exist from independent submitters. Trust is emerging but not yet stable.
Proceed with value caps or geo limits
EMERGING
Effective evidence ≥ 10.0. Multiple independent submitters, good graph health, consistent behavioral history across receipt types.
Proceed with standard policy thresholds
CONFIDENT
Effective evidence ≥ 25.0. Deep receipt history, high submitter diversity, strong graph health, no unresolved disputes. Trust is robust across categories.
Proceed. Eligible for elevated permissions
Key insight: effective evidence = Σ(receipt_weight) where receipt_weight = submitter_credibility × time_decay × graph_health × provenance_tier. A single host-verified receipt from a credible submitter counts more than 100 self-asserted receipts from new accounts.
// Sybil Resistance
Why lying hurts the liar

How effective evidence works

Raw receipt count is not trust. Each receipt is multiplied by four factors before contributing to confidence:

receipt_weight =
submitter_credibility // 0.0 – 1.0
× time_decay // recent = higher
× graph_health // closed loops = lower
× provenance_tier // host-verified = 1.0×

A new, unverified submitter with no history starts at 0.1×. Even 1,000 receipts from such submitters only contributes 100 effective evidence units — equivalent to just 4 host-verified receipts.

What happens when you attack

SUBMITTING FAKE RECEIPTS
Each fake receipt from a low-credibility account contributes ~0.04 effective evidence. The attacker spends real infrastructure cost for near-zero trust gain. Graph analysis also flags the pattern, reducing graph_health for the entire cluster.
FILING FALSE REPORTS
Dismissed reports lower your submitter_credibility. After multiple dismissed reports, your legitimate receipts are down-weighted too — making your own trust profile harder to build.
THE ASYMMETRY
Legitimate actors accumulate credibility over time. Attackers burn credibility faster than they can create fake accounts. The cost of a successful attack grows exponentially with each attempt.
